01目的と位置づけ

何のためのデータベースか

決算チェックシート（決算チェック_YYYYMMDD.xlsx）が数値の時系列蓄積に特化しているのに対し、Earnings Highlights DBは定性情報を含む決算ハイライトの記録と横断分析を担う。

  • Beat/Miss の数値サマリー（EPS、Revenue、EBITDA）
  • ガイダンスの方向と市場予想との乖離
  • 経営陣のコメント、戦略変更、課題
  • マーケット反応（株価変動、アナリスト評価）
  • 決算チェックシート更新時の技術メモ
設計思想 — なぜxlsxと分離するか
  1. リスク回避：8,700行×60列・36,000数式のxlsxを不必要に触る回数を減らす
  2. 情報の性質が違う：数値はセル、定性情報はテキスト。フォーマットが根本的に異なる
  3. 運用サイクルが違う：数値は一発入力、ハイライトは継続的に参照・振り返りする

04サマリーCSV 仕様

全30カラム、7つのカテゴリで構成。詳細は summary-columns.md を参照。

カテゴリ数主要カラム用途
識別情報5ticker, company, quarter, earnings_date, sector基本情報
実績 vs コンセンサス9eps/revenue/ebitda の actual, consensus, beat_pctBeat/Miss の定量評価
成長率（YoY）3eps_yoy_pct, revenue_yoy_pct, ebitda_yoy_pct前年同期比
マージン・CF4op_margin_pct/chg_bps, fcf_margin_pct/chg_bps収益性の変化
ガイダンス5guidance_eps, guidance_direction 等将来見通し
財務健全性1net_leverageBS健全度
定性タグ・評価4tags, sentiment, stock_reaction_pct, summary検索・総合評価

beat_pct の符号ルール

正（+） = Beat、負（−） = Miss

計算式：(actual - consensus) / |consensus| × 100

ガイダンスの guidance_*_vs_consensus_pct も同じ符号ルール。

tags カラムの運用

セミコロン区切りのフリーテキスト。頻出タグ例：

カテゴリタグ例
経営CEO交代, CFO交代, 組織再編, リストラ
戦略ガイダンス撤回, 上方修正, 自社株買い, M&A
財務デレバレッジ, コスト規律, マージン改善, 減損
事業新製品, 市場拡大, 競争激化, 規制リスク

※ 固定リストではなく自然蓄積。同じ概念に異なるタグを使わないよう過去タグを確認してから付与。

sentiment の基準

値基準典型パターン
Positive実績Beat＋ガイダンス上方修正EPS・Rev両方Beat、ガイダンス引き上げ
Mixed良い面と悪い面が混在EPS Beat だが Rev Miss、ガイダンス下方
Negative実績Miss＋ガイダンス下方修正EPS・Rev両方Miss、ガイダンス撤回

タイミング
Step 1（xlsx）は決算発表直後にClaudeチャットで実施。Step 2〜3 はその流れで続行。Markdown作成とCSV追記はClaudeが行い、GitHub PushはWeb UIから手動。

四半期の区切りでやること

  • 新しいサマリーCSVを作成（例：summary-2026Q1.csv）
  • 前四半期のCSVはそのまま保存（上書きしない）
  • カラム定義に変更があれば summary-columns.md も同時更新

07横断分析の方法

サマリーCSVに対して、Claudeに日本語で質問するだけで横断分析ができる。

数値ベースの抽出

  • 「2025Q4でEPSとRevenueの両方Beatした銘柄は？」
  • 「ガイダンスが上方修正だった銘柄を一覧で」
  • 「マージン改善が200bps以上だったのは？」
  • 「株価反応が良かった順にソートして」

定性ベースの抽出

  • 「CEO交代があった銘柄は？」 → tags で検索
  • 「コスト改善が目立った銘柄の詳細を見せて」 → CSV → 該当MDを参照
  • 「sentimentがNegativeだった銘柄の共通点は？」

複合条件

  • 「EPS Beatだがガイダンス下方修正で株価が下がった銘柄」
  • 「Revenue成長率トップ5のうちマージンも改善していたのは？」
分析の流れ
粗い絞り込みはCSVで行い、深掘りは個別Markdownを参照する2段階が基本パターン。

08GitHub操作手順

新規ファイルの追加（Markdown）

  1. earnings-insight リポの highlights/ フォルダに移動
  2. Add file → Upload files
  3. Claudeから受け取った .md ファイルをドラッグ&ドロップ
  4. Commit changes

CSVの更新（行追加）

方法A：ファイル丸ごと差し替え（推奨）

  1. Claudeから更新済みCSV全文を受け取る
  2. GitHub上で CSV を開く → 鉛筆アイコン（Edit）
  3. 全選択 → 新しい内容で置換 → Commit changes

方法B：末尾に1行追加

  1. GitHub上で CSV を開く → Edit
  2. 最終行の後にClaude出力の1行をペースト → Commit changes

※ 方法Aが確実。列ズレを防げる。

新しいフォルダの作成

GitHub Web UIでは空フォルダ作成不可。Add file → Create new file でファイル名欄に フォルダ名/ファイル名 と入力すると同時作成される。

CSVのカンマ・改行に注意
summary や tags 内にカンマがある場合、値全体を "..." で囲む。Claude出力は処理済みだが、手動編集時は注意。

09運用ルール・注意点

ルール 1 — xlsxとHighlightsは独立運用
決算チェックシート（xlsx）とHighlights DB（CSV + MD）は別系統。xlsxの運用ルール（数式保護、LibreOffice経由行挿入等）はHighlightsに無関係。逆も同様。
ルール 2 — CSVヘッダー行は変更しない
1行目はカラム定義と一致が必要。カラム追加・変更時は summary-columns.md も同時更新。
ルール 3 — 過去CSVは上書きしない
summary-2025Q4.csv はシーズン終了後もそのまま保存。新四半期は新CSVファイルを作成。
ルール 4 — 不明点は自己判断しない
不明点はMarkdownに「要確認」と明記して保留。推測値を入れない。決算チェック鉄則3と同じ精神。
ルール 5 — tagsは自然蓄積
事前に完全定義せず銘柄を重ねて蓄積。同一概念の表記揺れを避けるため過去タグを確認してから付与。
ルール 6 — MarkdownはGitHub上で閲覧
GitHubの自動レンダリングを利用。HTML化、GitHub Pages追加、ランチャー更新はいずれも不要。

Markdownセクション6が橋渡し

各銘柄Markdownのセクション6「xlsx反映メモ」は、xlsx更新時に発見した技術的注意点（セル書式問題、Diluted Shares選択、Gross Margin計算方法等）を記録する場所。次回同じ銘柄の決算時に前回の教訓を即座に参照できる。
